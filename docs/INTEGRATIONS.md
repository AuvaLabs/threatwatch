# ThreatWatch Integrations

How to consume ThreatWatch from Microsoft Teams, Azure Logic Apps, Power Automate,
Slack, Discord, SIEMs, and any HTTP-aware tool. Every endpoint is plain JSON
(or RSS/STIX where noted), gzip-compressed, ETag-aware, and CORS-enabled.

> **Base URL** in the recipes below: `https://threatwatch.auvalabs.com`
> If you self-host, swap in your own host:port (default `8098`).

---

## Quick start

Pull the global daily briefing and post the headline:

```bash
curl -s https://threatwatch.auvalabs.com/api/briefing \
  | jq -r '"[\(.threat_level)] \(.headline // .what_happened[0:140])"'
```

Pull all articles published in the last hour:

```bash
SINCE=$(date -u -d '1 hour ago' +%FT%TZ)
curl -s "https://threatwatch.auvalabs.com/api/since?ts=${SINCE}&limit=100" \
  | jq '.articles | length'
```

## Endpoint reference

All endpoints are read-only HTTP `GET`, no auth required, safe to poll from
public Logic Apps / Power Automate flows. The two write endpoints
(`/api/feedback`, `POST /api/watchlist`) are token-protected and meant for
self-hosted deployments only.

### Briefings

| Endpoint | Use case | Refresh | Notable fields |
|---|---|---|---|
| `GET /api/briefing` | Daily global TL;DR | Every ~30 min | `threat_level`, `headline`, `what_happened`, `what_to_do`, `outlook`, `source_articles[]` |
| `GET /api/briefing/na` | North America digest | Every ~30 min | Same shape, regional scope |
| `GET /api/briefing/emea` | EMEA digest | Every ~30 min | Same shape, regional scope |
| `GET /api/briefing/apac` | APAC digest | Every ~30 min | Same shape, regional scope |
| `GET /api/top-stories` | AI-curated headlines | Every ~30 min | `stories[]` with `headline`, `summary`, `significance`, `link` |

### Articles & search

| Endpoint | Use case | Refresh |
|---|---|---|
| `GET /api/articles?limit=N&offset=N` | Paginated article corpus | Every ~10 min |
| `GET /api/since?ts=<ISO8601>&limit=N` | Incremental fetch since timestamp — best for alerting | Every 5–10 min |
| `GET /api/cve/<CVE-ID>` | All articles referencing a CVE | On-demand |
| `GET /api/rss` | RSS 2.0 feed of confirmed cyber attacks | Outlook/Feedly polls every 15–60 min |

### Intelligence surfaces

| Endpoint | Use case |
|---|---|
| `GET /api/clusters` | Incident clusters grouped by entity (CVE / actor / org) with AI synthesis |
| `GET /api/actor-profiles` | APT/criminal-group profiles with observed ATT&CK techniques |
| `GET /api/campaigns` | Persistent threat campaigns with stable UUIDs |
| `GET /api/campaign/<id>` | Single campaign detail |
| `GET /api/trends` | Spike detection (today vs 14d baseline) + 7d/30d top-mentioned leaderboard |

### Standards-based exports

| Endpoint | Format | SIEM/SOAR use case |
|---|---|---|
| `GET /api/stix` | STIX 2.1 bundle | Direct ingest into Microsoft Sentinel, Splunk, Elastic, OpenCTI, MISP |
| `GET /api/rss` | RSS 2.0 XML | Outlook RSS subscriber, Feedly, Slack RSS app |

### Operational

| Endpoint | Use case |
|---|---|
| `GET /api/health` | Liveness probe (`status: ok\|degraded\|stale`) |
| `GET /api/stats` | Last 10 pipeline runs (articles fetched, dedup ratio, AI cost) |
| `GET /api/quality` | Per-feed health, classification quality metrics |

### Caching & politeness

- Every JSON endpoint sends an `ETag`. Send it back as `If-None-Match` to get a
  cheap `304 Not Modified`.
- Server-side cache TTL is 5 seconds — polling faster than that buys nothing.
- Suggested cadences: alerting flows = 5 min; dashboards = 2 min; daily
  briefing reposts = once or twice a day.

## Recipe 1 — Daily briefing to Microsoft Teams (Azure Logic App)

Posts the global briefing as an adaptive card to a Teams channel every morning.

**Trigger:** Recurrence — once per day at 08:00 local time.

**Action 1 — HTTP:**

```
Method:  GET
URI:     https://threatwatch.auvalabs.com/api/briefing
Headers: Accept: application/json
```

**Action 2 — Parse JSON** (paste the response from Action 1, schema can be
auto-generated by Logic App from a sample payload).

**Action 3 — Post adaptive card in a chat or channel** (Teams connector).
Use this card body, with `@{body('Parse_JSON')?['field']}` substitutions for
the dynamic content:

```json
{
  "type": "AdaptiveCard",
  "$schema": "http://adaptivecards.io/schemas/adaptive-card.json",
  "version": "1.4",
  "body": [
    {
      "type": "TextBlock",
      "text": "ThreatWatch — @{utcNow('yyyy-MM-dd')}",
      "weight": "Bolder",
      "size": "Medium"
    },
    {
      "type": "TextBlock",
      "text": "@{body('Parse_JSON')?['threat_level']}",
      "weight": "Bolder",
      "color": "@{if(equals(body('Parse_JSON')?['threat_level'], 'CRITICAL'), 'Attention', if(equals(body('Parse_JSON')?['threat_level'], 'ELEVATED'), 'Warning', 'Good'))}",
      "spacing": "Small"
    },
    {
      "type": "TextBlock",
      "text": "@{body('Parse_JSON')?['headline']}",
      "wrap": true,
      "weight": "Bolder",
      "size": "Large"
    },
    {
      "type": "TextBlock",
      "text": "@{body('Parse_JSON')?['what_happened']}",
      "wrap": true,
      "spacing": "Medium"
    }
  ],
  "actions": [
    {
      "type": "Action.OpenUrl",
      "title": "Open ThreatWatch",
      "url": "https://threatwatch.auvalabs.com"
    }
  ]
}
```

The threat-level colour mapping: `CRITICAL → Attention (red)`,
`ELEVATED → Warning (amber)`, anything else → `Good (green)`. If `headline`
is empty for an older briefing, swap it for `substring(body('Parse_JSON')?['what_happened'], 0, 140)`.

## Recipe 2 — New IOC / article alerts to Teams or Slack

Polls `/api/since` every 5 minutes and posts only what's new.

**Trigger:** Recurrence — every 5 minutes.

**Action 1 — Initialize a `lastTs` variable** persisted across runs (use a Logic
App variable file, Azure Table, or a SharePoint list cell). On first run,
default to `addMinutes(utcNow(), -30)`.

**Action 2 — HTTP:**

```
Method: GET
URI:    https://threatwatch.auvalabs.com/api/since?ts=@{variables('lastTs')}&limit=50
```

**Action 3 — For each article in `body('HTTP').articles`:**

Filter to high-confidence cyber-attack alerts to keep noise down:

```
@and(
  equals(item()?['is_cyber_attack'], true),
  greaterOrEquals(item()?['confidence'], 80)
)
```

Then post a Teams message (or Slack `chat.postMessage`) with the article
title, source, category, CVSS score (if any), and link.

**Action 4 — Update `lastTs` to `body('HTTP').next_cursor`** so the next run
picks up exactly where this one left off. The endpoint also returns
`total_new`, `truncated`, and `count` so you can detect backlogs.

## Recipe 3 — Critical CVE webhook to Slack / Discord / generic

ThreatWatch already has a built-in webhook dispatcher (`modules/webhook.py`)
that fires on briefing threat-level changes. Set these env vars in `.env`:

```bash
WEBHOOK_URL=https://hooks.slack.com/services/T0000/B0000/xxxxxxxx
WEBHOOK_BRIEFING_MIN_LEVEL=ELEVATED   # CRITICAL | ELEVATED | MODERATE | GUARDED | LOW
WEBHOOK_BRIEFING_COOLDOWN_HOURS=6
```

Auto-detected payload shape: Slack incoming webhook, Discord webhook, or
generic JSON (one POST per fired event). Deduplication is on `(level, day)`
plus a 6-hour cooldown to prevent spam.

The same endpoint works with Teams Incoming Webhook URLs — paste the Teams
webhook URL into `WEBHOOK_URL` and the dispatcher picks the generic-JSON
shape. For richer Teams cards, prefer Recipe 1 (Logic App).

## Recipe 4 — STIX 2.1 ingest into SIEM

`GET /api/stix` returns a STIX 2.1 bundle suitable for direct ingest into
threat-intelligence platforms.

- **Microsoft Sentinel:** Use the *Threat Intelligence — TAXII* data connector
  pointed at this URL (TAXII 2.1 server adapter required if your tenant doesn't
  accept raw STIX bundles — wrap with [opentaxii](https://github.com/eclecticiq/OpenTAXII)
  or similar).
- **Splunk:** Use the Splunk Add-on for STIX/TAXII or a custom modular input
  scheduled every 30 min.
- **Elastic:** Use the *threatintel* integration with the *misp* or *otx*
  module configured to fetch the bundle URL.
- **OpenCTI / MISP:** Both natively understand STIX 2.1 — register
  `https://threatwatch.auvalabs.com/api/stix` as a remote feed.

## Recipe 5 — Telegram channel (built-in)

ThreatWatch ships a first-class Telegram dispatcher (`modules/telegram.py`)
that posts the global briefing to a channel via the Bot API. Two delivery
modes are supported and can be combined:

**Mode A — alert on threat-level change (in-process):** Same dedup logic as
the Slack/Discord webhook (level-change OR cooldown lapsed). Set these in
`.env`:

```bash
TELEGRAM_BOT_TOKEN=123456:AA-...           # from @BotFather
TELEGRAM_CHAT_ID=@your_channel             # or -100123456789 for private
TELEGRAM_MIN_LEVEL=ELEVATED                # CRITICAL | ELEVATED | MODERATE | …
TELEGRAM_COOLDOWN_HOURS=6
TELEGRAM_DASHBOARD_URL=https://threatwatch.auvalabs.com
```

The dispatcher fires automatically inside `ai_enrichment.run_ai_enrichment`
right after the briefing is generated. Independent state file
(`data/state/telegram_briefing_last_alert.json`) — fully decoupled from the
existing webhook dedup, so an org can drive both Slack and Telegram in
parallel without one path muting the other.

**Mode B — guaranteed daily post (system cron):** When you want a once-a-day
post regardless of threat level, use the standalone script. Add to your
host crontab:

```bash
# Post the latest briefing to Telegram every day at 08:00 UTC
0 8 * * * /home/deploy/threatwatch/scripts/post_briefing_to_telegram.py
```

The script exits 1 on failure so cron MAILTO catches delivery problems.
It does not consume the alert dedup state, so Mode A and Mode B can both
fire on the same briefing without interfering.

**Bot setup (one time):**
1. Message `@BotFather` on Telegram, run `/newbot`, follow the prompts → save the token.
2. For a public channel, your `chat_id` is `@channel_handle`. For a private
   channel/group: add the bot, send any message, then call
   `https://api.telegram.org/bot<TOKEN>/getUpdates` and copy the numeric
   `chat.id` from the response.
3. Make the bot a channel admin (so it can post). Permissions needed: *Post Messages*.

**Message format:** Telegram-compatible HTML (escaped). Includes the level
badge, the LLM-written `headline` if present, the truncated `what_happened`
narrative, up to three priority actions, and a "Open dashboard →" link.
Capped at 4000 chars to stay safely under Telegram's 4096 limit.

## Recipe 6 — RSS in Outlook / Feedly / Slack

`GET /api/rss` is a standard RSS 2.0 feed of every article that survived
the pipeline (dedup + enrichment). Each item carries title, link, category,
summary, and publish date. Filter further on the consumer side if you only
want a subset (e.g. by category or keyword).

- **Outlook desktop:** *RSS Feeds* folder → right-click → *Add a New RSS Feed*
  → paste `https://threatwatch.auvalabs.com/api/rss`.
- **Feedly:** *Add Source* → paste the URL.
- **Slack:** `/feed subscribe https://threatwatch.auvalabs.com/api/rss` in
  the target channel.

Refresh interval suggestion: 15–60 min. The pipeline writes a fresh feed
every ~10 min so anything tighter is wasted polling.

## Auth, rate limits, and CORS

- **Auth:** None on read endpoints. The two write endpoints
  (`POST /api/watchlist`, `POST /api/feedback`) require
  `Authorization: Bearer <token>` matching the `WATCHLIST_TOKEN` env var
  (one shared token gates both writes) and are intended for self-hosted
  deployments where you own the secret. If `WATCHLIST_TOKEN` is unset the
  writes refuse fail-closed.
- **Rate limit:** No hard limit on the demo host. Be polite — a 5-second
  server-side cache means anything faster than ~10/min is wasted. Set a
  user-agent that identifies your tool so we can reach out if your flow
  causes issues.
- **CORS:** `Access-Control-Allow-Origin: *` on all `/api/*` routes except
  `/api/health` and `/api/watchlist` (same-origin only).

## Self-hosting

If you'd rather not depend on the demo host, ThreatWatch is a single
`docker compose up` away on any Linux box with 2GB RAM. See the main
[README](../README.md#quick-start) for setup. Once running, swap
`https://threatwatch.auvalabs.com` for `http://your-host:8098` in every
recipe above — the API surface is identical.
